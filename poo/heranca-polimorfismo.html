Herança e Polimorfismo em C#

Programação Orientada a Objetos

Imagine que nosso sistema de gerenciamento de veículos precise realizar acelerações de acordo com o tipo do veículo. Se o veículo for uma Moto, ela consome apenas 1 de combustível por aceleração e ganha 2 de velocidade. Se for um Carro (ou veículo genérico), consome 2 de combustível e ganha 1 de velocidade.

Para implementar essa regra de negócio, poderíamos colocar um if no método Acelerar:

public void Acelerar()
{
    if(this.Tipo == ???????????) // Moto
    {
        this.QuantidadeCombustivel -= 1;
        this.Velocidade += 2;
    }
    else // Carro ou genérico
    {
        this.QuantidadeCombustivel -= 2;
        this.Velocidade += 1;
    }
}

Poderíamos criar um atributo Tipo na classe Veiculo, onde o número 1 representaria "Moto" e 2 "Carro".

A implementação seria algo como:

public class Veiculo
{
    public string Placa { get; set; }
    public int QuantidadeCombustivel { get; private set; }
    public int Velocidade { get; private set; }
    public string Modelo { get; set; }
    public int Tipo { get; set; }

    public void Acelerar()
    {
        if(this.Tipo == 1) // Moto
        {
            this.QuantidadeCombustivel -= 1;
            this.Velocidade += 2;
        }
        else // Carro ou genérico
        {
            this.QuantidadeCombustivel -= 2;
            this.Velocidade += 1;
        }
    }

    public void Abastecer()
    {
        this.QuantidadeCombustivel += 1;
    }
}

Veja que uma simples regra de negócio como essa fez nosso código crescer muito. E poderia ser pior: imagine se nosso sistema tivesse 10 tipos de veículos diferentes. Esse if seria maior ainda.

Precisamos encontrar uma maneira de fazer com que a criação de novos tipos de veículo não implique em um aumento de complexidade.

Reaproveitando código com a Herança

Uma solução seria ter classes separadas para Veiculo (o genérico/Carro) e Moto:

public class Veiculo
{
    public string Placa { get; set; }
    public int QuantidadeCombustivel { get; private set; }
    public int Velocidade { get; private set; }
    public string Modelo { get; set; }

    public void Acelerar()
    {
        this.QuantidadeCombustivel -= 2;
        this.Velocidade += 1;
    }

    public void Abastecer()
    {
        this.QuantidadeCombustivel += 1;
    }
}

public class Moto
{
    public string Placa { get; set; }
    public int QuantidadeCombustivel { get; private set; }
    public int Velocidade { get; private set; }
    public string Modelo { get; set; }

    public void Acelerar()
    {
        this.QuantidadeCombustivel -= 1;
        this.Velocidade += 2;
    }

    public void Abastecer()
    {
        this.QuantidadeCombustivel += 1;
    }
}

Ambas as classes possuem código bem simples, mas agora o problema é outro: a repetição de código entre ambas as classes. Se amanhã precisarmos guardar "Cor", por exemplo, precisaremos mexer em todas as classes que representam um veículo no sistema. Isso pode ser trabalhoso.

A ideia é, portanto, reaproveitar código. Veja que, no fim, uma Moto é um Veiculo, pois ambos têm Placa, QuantidadeCombustivel, Velocidade e Modelo. A única diferença é o comportamento no momento da aceleração. Podemos falar que uma Moto é um Veiculo: 

public class Moto : Veiculo
{

}

Quando uma classe é definida com :, dizemos que ela herda da outra (Veiculo) e por isso ela ganha todos os atributos e métodos da outra classe. Por exemplo, se Moto herdar de Veiculo, ela terá Placa, QuantidadeCombustivel, Velocidade, Acelerar() e Abastecer() automaticamente, sem precisar fazer nada. Dizemos que Moto é uma subclasse ou classe filha de Veiculo, e que Veiculo é a classe base ou classe pai. Veja o código a seguir: 

// Arquivo Moto.cs
public class Moto : Veiculo
{

}

// Código no formulário que utiliza a Moto
Moto m = new Moto();
m.Abastecer(); // método herdado de Veiculo

Basta usar a notação : e o C# automaticamente herda os métodos e atributos da classe pai. Mas a Moto tem o comportamento de Acelerar() diferente. Para isso, basta reescrever o comportamento na classe filha, usando a palavra override e marcando o método na classe pai como virtual, para indicar que ele pode ser sobrescrito: 

// Arquivo Veiculo.cs
public class Veiculo
{
    public virtual void Acelerar()
    {
        this.QuantidadeCombustivel -= 2;
        this.Velocidade += 1;
    }

    // Resto do código da classe
}

// Arquivo Moto.cs
public class Moto : Veiculo
{
    public override void Acelerar()
    {
        this.QuantidadeCombustivel -= 1;
        this.Velocidade += 2;
    }
}

// Código do formulário da aplicação
Moto m = new Moto();

// Chama o comportamento escrito no pai (Veiculo)
// QuantidadeCombustivel termina em 1 depois dessa linha
m.Abastecer();

// Chama o comportamento escrito na Moto
// QuantidadeCombustivel termina em 0, Velocidade termina em 2
m.Acelerar();

Veja nesse código que invocamos tanto Abastecer() quanto Acelerar(). No abastecimento, como a classe filha não redefiniu o comportamento, o método escrito na classe pai será utilizado.

Já na aceleração, o comportamento usado é o que foi sobrescrito na classe filha.

Atenção: o código anterior ainda não compila. Repare que o método Acelerar() da Moto manipula QuantidadeCombustivel e Velocidade. Mas essas propriedades são private! Atributos privados só são visíveis para a classe que os declarou. Os filhos não enxergam.

Queremos proteger nossos atributos mas não deixá-los privados nem públicos. Queremos protegê-los o suficiente para ninguém de fora acessar, mas permitir que quem herda tenha acesso. Para resolver, alteraremos o modificador de acesso para protected. Atributos/métodos marcados como protected são visíveis apenas para a própria classe e para as classes filhas: 

public class Veiculo
{
    public string Placa { get; set; }
    public int QuantidadeCombustivel { get; protected set; }
    public int Velocidade { get; protected set; }
    public string Modelo { get; set; }

    // ...
}

A classe Veiculo ainda pode ser instanciada sem problemas:

Veiculo v = new Veiculo();
v.Abastecer();

Veja que com herança conseguimos simplificar e reutilizar código ao mesmo tempo. A herança é um mecanismo poderoso mas deve ser utilizado com cuidado.

Reaproveitando a implementação da classe base

Quando fazemos a sobrescrita de métodos, muitas vezes queremos apenas estender o comportamento da classe base, aproveitando o que já foi implementado. Nessas situações podemos reutilizar o código do pai com a palavra base.

Imagine, por exemplo, que ao abastecer uma Moto, além de adicionar combustível, ela também precisa parar completamente (pois o piloto desce para abastecer). Sem base, duplicaríamos a lógica: 

// Sem base — duplicando a lógica de adicionar combustível
public class Moto : Veiculo
{
    public override void Abastecer()
    {
        this.QuantidadeCombustivel += 1; // código repetido do pai!
        this.Velocidade = 0;             // comportamento específico da Moto
    }
}

Com base, podemos chamar o método da classe pai e apenas adicionar o comportamento extra:

// Com base — reutilizando o código do pai
public class Moto : Veiculo
{
    public override void Abastecer()
    {
        base.Abastecer(); // QuantidadeCombustivel += 1 (reaproveita o código do pai)
        this.Velocidade = 0; // ao abastecer, a Moto para completamente
    }
}

Com essa implementação, o Abastecer da Moto delega ao método da classe base a adição de combustível e, em seguida, realiza o comportamento específico da moto. Se a lógica de abastecimento em Veiculo mudar no futuro, a Moto se beneficia automaticamente.

Repare que, como a classe filha não acessa mais QuantidadeCombustivel diretamente no Abastecer, essa propriedade poderia voltar a ser private — desde que o método Acelerar da Moto também usasse base. Porém, como o Acelerar da Moto ainda modifica QuantidadeCombustivel e Velocidade com valores diferentes dos do pai, ambas as propriedades precisam continuar como protected.

Polimorfismo

Nosso sistema possui um Radar que registra os veículos que passam por ele e acumula a velocidade total detectada. Começando com zero: 

public class Radar
{
    public int VelocidadeTotal { get; private set; }
}

Permitimos registrar um Veiculo e acumular sua velocidade:

public class Radar
{
    public int VelocidadeTotal { get; private set; }

    public void Registrar(Veiculo veiculo)
    {
        VelocidadeTotal += veiculo.Velocidade;
    }
}

Veiculo v1 = new Veiculo();
Veiculo v2 = new Veiculo();

Radar r = new Radar();
r.Registrar(v1);
r.Registrar(v2);

Ótimo. Mas o problema é que temos classes que representam diferentes tipos de veículos, e queremos registrar todos eles. Uma primeira solução seria ter um método Registrar() para cada classe específica: 

public class Radar
{
    public int VelocidadeTotal { get; private set; }

    public void Registrar(Veiculo veiculo) { /* ... */ }
    public void Registrar(Moto moto) { /* ... */ }
    public void Registrar(Carro carro) { /* ... */ }
    // mais um monte de métodos aqui!
}

Novamente caímos no problema da repetição de código.

Veja que Moto é um Veiculo. Isso é inclusive expresso através da relação de herança entre as classes. E já que Moto é um Veiculo, o C# permite que você passe uma Moto em lugares que aceitam referências do tipo Veiculo! 

public class Radar
{
    public int VelocidadeTotal { get; private set; }

    public void Registrar(Veiculo veiculo)
    {
        VelocidadeTotal += veiculo.Velocidade;
    }
}

Veiculo v1 = new Veiculo();
Moto m1 = new Moto();

Radar r = new Radar();
r.Registrar(v1);
r.Registrar(m1); // funciona! Moto é um Veiculo

O código funciona! O C# sabe que Moto herda todos os atributos e métodos de Veiculo, e portanto tem a certeza de que existe a propriedade Velocidade, podendo invocá-la sem problemas.

Agora, imagine que a Moto tenha um comportamento específico para emitir um som característico: 

class Moto : Veiculo
{
    public void EmitirSom()
    {
        // som específico da moto
    }
}

Se tentarmos chamar EmitirSom() dentro do Registrar:

public class Radar
{
    public int VelocidadeTotal { get; private set; }

    public void Registrar(Veiculo veiculo)
    {
        VelocidadeTotal += veiculo.Velocidade;
        veiculo.EmitirSom(); // não compila!
    }
}

O código não compila. Por quê? Porque o C# não consegue garantir que o que virá na variável veiculo será uma Moto. Naquela "porta" entra Veiculo ou qualquer filho de Veiculo. Portanto, tudo o que o C# consegue garantir é que o objeto que entrar ali tem tudo que Veiculo tem. Por isso, só podemos usar métodos e atributos definidos pelo tipo da variável (no caso, Veiculo).

Essa ideia de uma variável conseguir referenciar seu próprio tipo ou filhos desse tipo é conhecida como polimorfismo. Com o uso de polimorfismo, garantimos que a classe Radar funcionará para qualquer novo tipo de Veiculo que aparecer.

Se no futuro um novo tipo de veículo, como Caminhao, for criado, basta que ele herde de Veiculo e o Radar funcionará naturalmente, sem nenhuma modificação!

Exercícios

1. Qual a diferença entre private e protected?

  • Nenhuma. Em ambos, o atributo/método é visível para todos.
  • Só a própria classe enxerga atributos/métodos private, enquanto protected é visto pela própria classe e pelas classes filhas.
  • Só a própria classe enxerga atributos/métodos protected, enquanto private é visto pela própria classe e pelas classes filhas.

2. Para que serve a palavra override?

  • Para indicar que o método está sobrescrevendo um método da classe pai.
  • Para não permitir acesso ao atributo por outras classes.
  • Para indicar que esse método não deve ser utilizado.

3. Para que serve a palavra virtual?

  • Para permitir que o método seja sobrescrito pelas classes filhas.
  • Para indicar que esse método não deve ser sobrescrito.
  • Para sobrescrever um método da classe pai.
  • Para não permitir acesso ao atributo por outras classes.

4. Adicione a classe Moto na aplicação. Essa classe deve herdar de Veiculo e sobrescrever o comportamento do método Acelerar para que cada aceleração da moto consuma apenas 1 de QuantidadeCombustivel e aumente 2 em Velocidade.

Não se esqueça de utilizar as palavras virtual e override.

Repare que não precisamos modificar o tipo da variável veiculo, pois como Moto herda de Veiculo, podemos utilizar o polimorfismo para atribuir uma referência do tipo Moto em uma variável do tipo Veiculo.

Teste o botão de abastecimento — a Moto não declarou Abastecer, mas mesmo assim conseguimos invocá-lo, pois todo o comportamento da classe pai é herdado. Teste também a aceleração: como a Moto sobrescreve Acelerar, o comportamento especializado da moto será utilizado.

7. Crie a classe Carro dentro do projeto e faça com que ela herde da classe Veiculo.

O Carro é um veículo mais pesado e com tanque maior, portanto suas operações funcionam de forma diferente: cada Acelerar consome 3 de QuantidadeCombustivel e aumenta 1 em Velocidade; cada Abastecer adiciona 2 ao QuantidadeCombustivel (tanque maior). Ou seja, no Carro, precisaremos sobrescrever tanto o Acelerar quanto o Abastecer. Não se esqueça de usar virtual e override.

Para saber mais — o que é herdado?

Quando fazemos a classe Moto herdar da classe Veiculo, ela recebe automaticamente todos os atributos, propriedades e métodos da classe pai, porém os construtores da classe pai não são herdados. Se a classe filha precisar de um construtor que está na classe pai, ela deve explicitamente declará-lo em seu código.

Imagine, por exemplo, que para construir um veículo precisamos passar sua placa:

public class Veiculo
{
    public string Placa { get; set; }

    // Construtor sem argumentos
    public Veiculo() {}

    public Veiculo(string placa)
    {
        this.Placa = placa;
    }
}

Na classe Moto, como o construtor não é herdado, precisamos declará-lo explicitamente: 

public class Moto : Veiculo
{
    public Moto(string placa)
    {
        // A propriedade Placa veio herdada da classe Veiculo
        this.Placa = placa;
    }
}

Nesse construtor fizemos apenas a inicialização da placa, exatamente o mesmo código que temos no construtor da classe pai. Ao invés de repetirmos o código, podemos simplesmente chamar o construtor de Veiculo a partir do construtor de Moto utilizando a palavra base: 

public class Moto : Veiculo
{
    // Estamos chamando o construtor da classe pai que já faz a inicialização
    // da placa, por isso o corpo do construtor pode ficar vazio.
    public Moto(string placa) : base(placa) { }
}

Na verdade, dentro do C#, sempre que construímos uma instância de Moto, o C# sempre precisa chamar um construtor da classe Veiculo para fazer a inicialização da classe base. Quando não invocamos explicitamente o construtor da classe pai, o C# coloca implicitamente uma chamada para o construtor sem argumentos da classe pai: 

public class Moto : Veiculo
{
    // Nesse código o C# chamará o construtor sem argumentos de Veiculo.
    public Moto(string placa)
    {
        this.Placa = placa;
    }
}

Se a classe Veiculo não definir o construtor sem argumentos, teremos um erro de compilação se não invocarmos explicitamente um construtor da classe pai.
